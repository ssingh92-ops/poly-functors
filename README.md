# Poly: polynomial functors as containers in Lean 4

This repository develops a Lean 4 library for polynomial functors in the concrete container presentation. A polynomial functor is represented by a type of shapes together with, for each shape, a type of positions. If someone has never seen the language of polynomial functors before, a good intuition is that a shape describes the overall form of a data structure and the positions describe the slots inside that form where data can live. For example, a list-like container can be described by a natural number shape `n` together with positions `0,1,…,n-1`. In the library, the central object is `PolyC`, where `A` is the type of shapes and `B : A → Type` gives the positions for each shape.

Morphisms in the category are lens-style maps between containers. A morphism `f : P ⟶ Q` maps shapes forward from `P.A` to `Q.A`, and it maps positions backward from the target to the source. This is the standard “forward on shapes, backward on positions” pattern that makes container morphisms behave like lenses. The core file of the project defines these objects and morphisms, the identity morphism, composition, and extensionality lemmas that let later files prove equality of morphisms componentwise.

On top of the core category, the project builds increasingly rich categorical structure. The library includes semantics/evaluation machinery, sums and products, composition-style constructions, universal-property files, bridges to Mathlib’s `PFunctor`, and monoidal-style operators. The current milestone is an explicit equalizer construction. Given two parallel morphisms `f g : P ⟶ Q`, the equalizer object keeps exactly those shapes of `P` on which `f` and `g` agree at the shape level, and then quotients the position type by the equivalence relation generated by the two pullback maps on positions. This is the categorical idea that in the container presentation, equalizing shapes is a restriction, while equalizing positions is a quotient.

A major proof-engineering lesson from the project is that the equalizer quotient should depend only on the underlying shape `a : P.A` rather than the full subtype witness `⟨a, proof⟩` showing that `f` and `g` agree on shapes. If the quotient depends on the proof object itself, every later rewrite triggers fragile dependent transports. By moving the quotient to the underlying shape, the equalizer file isolates the remaining hard obligations to a small number of proof points rather than forcing transport problems throughout the development.

The intended public-facing module names are `Poly.Core`, `Poly.Equalizers`, and so on. In other words, the reusable library surface is the `Poly` namespace rather than a course-specific or user-specific prefix. The package configuration can still have any outer package name, but the library that others import is simply `Poly`. To make this work, the project root contains `Poly.lean` alongside a `Poly/` folder with files such as `Poly/Core.lean` and `Poly/Equalizers.lean`, and `lakefile.toml` declares `Poly` as a `lean_lib`. That is why users of the library can write `import Poly.Core` or `import Poly.Equalizers` without caring what the enclosing git repository is named.
