# Poly: polynomial functors as a Category in Lean 4

This repository develops a Lean 4 library for polynomial functors in the concrete **container** presentation, following the mathematical perspective of Nelson Niu and David Spivak’s *Polynomial Functors*. A polynomial functor is represented by a type of **positions** together with, for each position, a type of **directions**. If someone has not seen this language before, the basic idea is that a position describes one summand or “place” in the polynomial, while the corresponding direction type describes how data branches out from that position. In the library, the central object is `PolyC`, where `A` is the type of positions and `B : A → Type` gives the directions available at each position.

Morphisms in the category are **lens-style** maps between containers. A morphism `f : P ⟶ Q` maps positions forward from `P.A` to `Q.A`, and it pulls directions backward from the target to the source. This “forward on positions, backward on directions” pattern is one of the characteristic structural features of polynomial functors, and it is exactly what makes the morphisms behave like categorical lenses. The core file of the project defines these objects and morphisms, identity, composition, and extensionality lemmas used later to prove equality of morphisms componentwise.

On top of the core category, the project builds increasingly rich categorical structure. The library includes semantic/evaluation machinery, sums and products, composition-style constructions, universal-property files, bridges to Mathlib’s `PFunctor`, and monoidal-style operators. The current milestone is an explicit equalizer construction. Given two parallel morphisms `f g : P ⟶ Q`, the equalizer object keeps exactly those positions of `P` on which `f` and `g` agree, and then quotients the corresponding direction type by the equivalence relation generated by the two pullback maps on directions. In the container viewpoint, equalizing positions is a restriction, while equalizing directions is a quotient.

A major proof-engineering lesson from the project is that the equalizer quotient should depend only on the underlying position `a : P.A` rather than the full subtype witness `⟨a, proof⟩` showing that `f` and `g` agree on positions. If the quotient depends on the proof object itself, every later rewrite triggers fragile dependent transports. By moving the quotient to the underlying position, the equalizer file isolates the remaining hard obligations to a small number of proof points rather than forcing transport problems throughout the development.

The intended public-facing module names are `Poly.Core`, `Poly.Equalizers`, and so on. In other words, the reusable library surface is the `Poly` namespace rather than a course-specific or user-specific prefix. The package configuration can still have any outer package name, but the library that others import is simply `Poly`. That is why users of the library can write `import Poly.Core` or `import Poly.Equalizers` without caring what the enclosing Git repository is named.

## Relation to Mathlib

Mathlib already contains the unbundled notion of `PFunctor`. This project is not replacing that interface. Instead, it builds a book-guided categorical layer on top of the container viewpoint and then connects back to Mathlib through `PFunctorBridge.lean`. The contribution here is the extra categorical structure and file organization: explicit lens-style morphisms, semantic evaluation, algebraic constructors, universal-property development, and the current equalizer milestone.
